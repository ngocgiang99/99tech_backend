## Context

`step-03` ships a working write path with security guards and a stop-gap error mapping in each controller. There's no structured logging (Pino isn't wired), no request ID propagation, no metrics, no traces, no integration tests against real infrastructure, no layer-boundary lint enforcement, and no operational runbooks. The Story 1.19 GAP-05 (action-token rotation) is also unaddressed — the verifier from `step-03` reads only one secret.

This change is the "operationalization" change. After it, Epic 1 is structurally done: the system can be deployed, monitored, debugged, rotated, and tested against real infrastructure. The next change (`step-05`) starts Epic 2 (leaderboard cache + outbox), which is a different domain.

The change is moderately large because it touches many cross-cutting files, but each piece is mechanical:
- Pino setup is ~50 lines
- Error filter is ~80 lines
- Metrics module is ~150 lines (15 metric registrations from `architecture.md §12.1`)
- Tracing bootstrap is ~80 lines (mostly `@opentelemetry/*` import and config)
- ESLint boundaries config is ~50 lines
- Jest config is ~80 lines (two configs — unit and integration)
- The dual-secret verifier patch is ~20 lines
- The runbook is ~80 lines of prose

The risk is **scope creep**, not technical complexity. Sticking to the deliverables list above (and resisting "while we're here, let's also add X") is what keeps this change manageable.

## Goals / Non-Goals

**Goals:**
- Every HTTP request emits at least one structured JSON log line including `ts, level, msg, requestId, route, latencyMs, statusCode, userId(hashed)`. Secrets are redacted.
- Every error response has the envelope `{ error: { code, message, requestId, hint } }`. The `requestId` matches the `X-Request-Id` response header.
- `/metrics` (registered here, exposed in `step-07`) emits valid Prometheus text-format with all the metrics from `architecture.md §12.1`.
- Tracing is plumbed end-to-end: a request with an inbound `traceparent` header continues that trace; a request without one starts a new trace.
- `mise run lint` enforces the hexagonal boundary rules. A domain file importing from `@nestjs/common` causes a lint error.
- `mise run test:coverage` enforces a ≥80% threshold for lines, branches, functions, statements. Failure is a build failure.
- `mise run test:integration` runs Testcontainers-backed Postgres + Redis tests for the persistence and rate-limit adapters. The first time it runs, it pulls the necessary images; subsequent runs reuse them.
- The action-token rotation runbook documents a 4-step procedure (deploy with both secrets → rotate primary → wait for token TTL → remove prev). The `HmacActionTokenVerifier` reads `ACTION_TOKEN_SECRET_PREV` if set and tries it on signature failure.
- All controller-level error try/catch blocks from `step-03` are deleted.

**Non-Goals:**
- The `/health`, `/ready`, `/metrics` HTTP endpoints (the metrics REGISTRY exists here, but the controller exposing it is `step-07`).
- E2E tests against the full docker-compose stack — `step-07`.
- A log shipper or any Pino transport beyond `pino-pretty` for dev and stdout JSON for prod.
- Grafana dashboards (JSON file or otherwise).
- A NATS Testcontainer integration test — `step-06` is when NATS lands.
- Automated secret rotation — manual is the v1 deliverable per ADR / `IMPROVEMENTS.md` I-SEC-04.
- Distributed tracing across services — we plumb the SDK; whether traces traverse instances depends on infra (`OTEL_EXPORTER_OTLP_ENDPOINT`).

## Decisions

### Decision 1: Use `nestjs-pino` for the NestJS-Pino integration

**What**: Add `nestjs-pino` (a small NestJS-aware wrapper around Pino) as the primary logger. Configure it via `LoggerModule.forRoot({ pinoHttp: { ... }})` in `AppModule`, with custom request-ID extraction, body redaction, and a serializer for HTTP requests.

**Why**:
- **`nestjs-pino` is the established NestJS adapter**: it does the per-request child logger automatically and integrates with `app.useLogger(...)` so framework logs are also Pino-formatted.
- **Replaces the default NestJS console logger** without losing framework-internal log lines.
- **Fastify-aware**: it works with the Fastify adapter we use (the default-Express integration is also supported but irrelevant here).

**Alternatives considered**:
- **Manual Pino wiring** (no `nestjs-pino`). Rejected — duplicates the boilerplate `nestjs-pino` provides.
- **Winston via `nest-winston`**. Rejected — Pino is the choice in `architecture.md` ADR-13. Winston is heavier.
- **Roll our own structured logger**. Rejected — Pino is the standard for high-throughput Node.

### Decision 2: Request ID is a ULID, generated by a Fastify hook BEFORE NestJS

**What**: Register a Fastify `onRequest` hook in `main.ts` (BEFORE `app.listen`) that:
1. Reads the inbound `X-Request-Id` header. If present and matches `^[A-Za-z0-9]{16,40}$`, use it.
2. Otherwise, generate a fresh ULID via the `ulid` package.
3. Set `request.requestId = id`.
4. Set the `X-Request-Id` response header to `id`.
5. Bind a Pino child logger to `request.log = baseLogger.child({ requestId: id })`.

**Why**:
- **Fastify hooks run before NestJS DI**: this is the only place we can guarantee the request ID exists for the very first log line (the request-receipt log).
- **ULID over UUID**: ULIDs are sortable by time, useful for log queries. Same length as UUIDs but lexicographically time-ordered.
- **Trust inbound IDs**: a load balancer or upstream service may have already generated one; honoring it lets us trace across boundaries.

**Alternatives considered**:
- **NestJS middleware** for the request-ID. Rejected — runs AFTER Fastify hooks, so the request-receipt log would be missing the ID.
- **UUID v4** instead of ULID. Considered, but ULID's sortability is genuinely useful for log timelines.
- **No inbound trust** (always generate fresh). Rejected — breaks distributed tracing where an upstream LB has already assigned an ID.

### Decision 3: Global exception filter is a single class registered via `app.useGlobalFilters(...)`

**What**: `src/scoreboard/interface/http/error-filter.ts` exports `@Catch() class HttpExceptionFilter implements ExceptionFilter`. Registered in `main.ts` via `app.useGlobalFilters(new HttpExceptionFilter())`. The filter has a switch on the error type that maps to the documented HTTP code and builds the envelope.

**Why**:
- **Single point of error normalization**: every error path goes through one place.
- **Replaces the per-controller try/catch**: cleaner controller code, single source of truth for the envelope shape.
- **NestJS canonical pattern**: `@Catch()` filters are the documented way to globally handle errors.

**Alternatives considered**:
- **Multiple `@Catch(SpecificError)` filters**. Considered but rejected for v1 — the switch is clearer than fanning across files.
- **Middleware-based error handling**. Rejected — Fastify's error handling is awkward to plug into; NestJS filters are easier.

**Idempotent replay caveat**: The `IdempotencyViolationError` from the controller's success-path catch (where it's translated into a 200) does NOT go through the global filter. Only the unhandled errors do. The controller's local catch for `IdempotencyViolationError` STAYS — it's a success path, not an error path. The other try/catch logic (`InvalidArgumentError → 400`, etc.) is what gets deleted.

### Decision 4: `prom-client` is the metrics library; metrics live in `src/shared/metrics/`

**What**: Use `prom-client` to declare each metric as a module-level singleton. `src/shared/metrics/write-path-metrics.ts` exports `httpRequestDuration`, `scoreIncrementTotal`, `actionTokenVerifyTotal`, etc. — each with its labels declared up-front. Provide them via a NestJS `MetricsModule` that's `@Global()` so any service can `@Inject(SCORE_INCREMENT_TOTAL)` them.

**Why**:
- **`prom-client` is the de-facto Node Prometheus library**: stable, well-maintained, supports counter/gauge/histogram/summary.
- **Module-level singletons**: metrics MUST be created exactly once per process (re-creating a counter throws). A `@Global()` module enforces this.
- **Token-based DI for metrics**: each metric is a separate provider with a string token (`SCORE_INCREMENT_TOTAL`, etc.). Services inject only the metrics they need.

**Alternatives considered**:
- **OpenTelemetry metrics**. Considered, but OTel metrics is younger than Prometheus and the architecture (ADR-13) explicitly chose Prometheus exposition with OTel for traces only.
- **Manual `text/plain` metrics serialization**. Rejected — `prom-client` handles the format correctly.

### Decision 5: OpenTelemetry SDK is initialized in `tracing.bootstrap.ts`, called from `main.ts` BEFORE `NestFactory.create`

**What**: Tracing init must run before any framework imports happen so the auto-instrumentation can patch them. The pattern: `import { initTracing } from './shared/tracing/tracing.bootstrap'; await initTracing();` is the FIRST line of `main.ts`. Then `import { AppModule } from './app.module';` and `NestFactory.create(...)`.

**Why**:
- **Auto-instrumentation works by patching modules at import time**. If `pg` or `fastify` is imported before `initTracing()`, the patching is silent no-op.
- **`initTracing()` is a no-op when `OTEL_EXPORTER_OTLP_ENDPOINT` is unset**. Local dev without OTel infrastructure works as before.

**Alternatives considered**:
- **`-r` flag (`node --require`) to preload tracing**. Cleaner separation but adds another mise task config. Rejected for v1; revisit if the import-order constraint becomes painful.

### Decision 6: ESLint boundary plugin uses element types matching `README.md §11.2`

**What**: Configure `eslint-plugin-boundaries` with element types:
- `domain` (`src/scoreboard/domain/**`) — depends on nothing
- `application` (`src/scoreboard/application/**`) — depends on `domain` only
- `infrastructure` (`src/scoreboard/infrastructure/**`) — depends on `application` + `domain`
- `interface` (`src/scoreboard/interface/**`) — depends on `application`
- `shared` (`src/shared/**`) — usable by anyone

The plugin's `boundaries/element-types` rule fires `error` on violation.

**Why**:
- **Direct mapping from README §11.2** — no ambiguity about what's allowed.
- **Lint-time enforcement**: violations show up in the dev's editor and in CI. Catches the "domain accidentally imports `@nestjs/common`" mistake at write time.
- **`error` not `warn`**: boundary violations are bugs, not style preferences.

**Alternatives considered**:
- **`eslint-plugin-import` with custom rules**. Rejected — `eslint-plugin-boundaries` is the purpose-built tool.
- **A custom AST checker via `ts-morph`**. Rejected — overkill.

### Decision 7: Jest has TWO configs — unit and integration — driven by separate mise tasks

**What**:
- `jest.config.ts`: matches `test/unit/**/*.test.ts`, no setup, fast feedback. Runs via `mise run test`.
- `jest.integration.config.ts`: matches `test/integration/**/*.test.ts`, includes Testcontainers setup, slower. Runs via `mise run test:integration`.
- `mise run test:coverage` runs the unit config with `--coverage --coverageThreshold='{global:{lines:80,branches:80,functions:80,statements:80}}'`.

**Why**:
- **Unit and integration have very different speed profiles**: unit tests run in ~3 seconds; integration tests pull Docker images and take ~30 seconds. Splitting them lets devs iterate fast on unit tests and run integration on PR.
- **Coverage gate runs only on the unit suite** (since unit tests are deterministic and complete; integration tests are slower and broader).
- **`mise run check` (the pre-push gate)** runs `lint + typecheck + test` (unit only). CI runs `lint + typecheck + test:coverage + test:integration`.

**Alternatives considered**:
- **Single config with `--testPathPattern`**. Rejected — can't have different setup files cleanly.
- **Vitest**. Rejected — `architecture.md` and `mise.toml` already chose Jest.

## Risks / Trade-offs

- **[Risk]** `nestjs-pino` adds a layer between NestJS's logger and Pino. Custom log lines from inside services need to be aware of this (`@InjectLogger() logger: PinoLogger` instead of the default NestJS `Logger`) → **Mitigation**: document the import pattern in design.md and provide a single example in the runbook. The DI is the only ergonomic difference.

- **[Risk]** Auto-instrumentation patches `pg` at import time. If `pg` is imported BEFORE `initTracing()` (e.g. by some upstream module that loaded earlier), the patching silently does nothing → **Mitigation**: put `initTracing()` as the very first non-import statement in `main.ts`. Document the constraint at the top of `tracing.bootstrap.ts`.

- **[Risk]** Testcontainers requires Docker to be running locally. CI environments without Docker can't run integration tests → **Mitigation**: Testcontainers is the spec choice (`architecture.md` Step 5). CI must provide Docker. If Docker is unavailable, the integration suite is skipped and the dev runs unit + lint + typecheck only.

- **[Risk]** `mise run test:coverage` enforces 80% globally, but the domain layer is supposed to hit 100% per `step-02`'s spec. If domain coverage drops, the global gate doesn't catch it (global average can be 80%+ even if domain is 60%) → **Mitigation**: the coverage threshold can also be set per-directory in `jest.config.ts` (`coverageThreshold: { 'src/scoreboard/domain/**/*.ts': { lines: 100 } }`). Add this in Task 9 of tasks.md.

- **[Risk]** The dual-secret verifier introduces a subtle bug surface: if both secrets are valid for the same token (e.g. unintentional secret reuse), the verifier accepts ambiguously → **Mitigation**: the verifier tries primary FIRST and only falls through on signature failure. There's no scenario where both secrets are valid for the same token unless they're identical (which would be a config error caught earlier).

- **[Risk]** The runbook is operator-facing prose; it can drift from the verifier code → **Mitigation**: the runbook has a section "Verification" where it tells the operator to run `curl -X POST .../v1/scores:increment` with a token signed by the prev secret and expect a 200. If that test fails, the runbook is wrong and needs updating.

- **[Trade-off]** `prom-client`'s `Counter`/`Histogram` types aren't sealed — anyone can `inc()` a counter by mistake. Mitigation: pass them through `@Inject(METRICS_TOKEN)` so the dependency is explicit. Accepted.

- **[Trade-off]** ESLint boundary checks add ~2 seconds to `mise run lint`. Worth it for the safety.

- **[Trade-off]** The 80% coverage gate is enforced for the WHOLE codebase, but `src/shared/{metrics,tracing}` is hard to unit-test (they're mostly initialization). Either exclude them from coverage or write thin tests. Decision: exclude `src/shared/{metrics,tracing}/**` from coverage and document why.

## Open Questions

- **Q1: Should `requestId` be exposed as a NestJS request-scoped provider?** So services can `constructor(@Inject(REQUEST_ID) private requestId: string) {}`. **Default decision**: yes — it's a 5-line provider and useful for any service that needs to log without going through the controller. Add it to `src/shared/logger/`.

- **Q2: How are histogram buckets configured?** `prom-client` defaults are `[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10]` seconds. **Default decision**: keep defaults for v1; revisit when load tests show actual latency distributions.

- **Q3: Should the `MetricsModule` register a `process.start` gauge?** Yes — add `scoreboard_process_start_time_seconds` set once at boot. Useful for grafana dashboards that show "uptime" or "deploy time".

- **Q4: How does the GAP-05 runbook handle the case where the operator deploys with `ACTION_TOKEN_SECRET_PREV` but forgets to set `ACTION_TOKEN_SECRET`?** **Default decision**: the zod schema requires `ACTION_TOKEN_SECRET` (no `.optional()`), so the boot fails at config parse. The prev-only deploy is impossible by construction. Document this in the runbook.

- **Q5 — DECISION-1**: What is the rollover window length for action-token rotation? The runbook tells the operator to "wait for the rollover window to close" between rotation steps. Options:
  - **(a) 5 minutes** — equal to the action-token TTL. Once the TTL expires, no in-flight tokens were signed by the prev secret. SAFEST and SHORTEST.
  - **(b) 15 minutes** — gives a generous grace period for tokens that may have been issued just before the rotation deploy and are still in flight (e.g. mobile clients with delayed retries).
  - **(c) 1 hour** — extra-conservative; covers any edge case like clock skew or paused processes.
  - **(d) Configurable** — make it an env var so different environments can pick.
  - **<DECISION>** — `/opsx:apply` will halt at the runbook authoring task and prompt for your choice. The runbook then bakes the chosen value into its "wait" instruction.
